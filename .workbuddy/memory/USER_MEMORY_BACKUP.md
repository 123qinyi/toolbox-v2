### 问题排查多维度思考法
- 遇到任何问题，必须从以下维度逐一排查，不能只盯着一个属性反复调整：
  - 外层容器 → 检查父级组件的默认样式（如 Card 的 gap、flex 布局等）
  - 内层元素 → 检查子组件的 padding / margin / spacing
  - 全局样式 → 检查是否有全局 CSS 覆盖
  - 浏览器默认 → 考虑浏览器自带样式的影响
  - 逐个验证 → 列出所有可能原因，从影响最大的开始验证
- 反例：只改 padding 反复试，没发现父级 gap-6 是真正根源
- **追加教训（2026-07-04）**：弹窗高度不生效，第一反应猜"三层布局瓜分空间"——错误。实际根因是内层 `max-h-[32vh]` 硬限死。规则：改样式不生效时，停止猜测，立刻从外层到内层逐行读代码，查是否有硬编码覆盖。不能只看外层一处就下结论。
- **追加教训（2026-07-07）**：flex 布局中子元素不应设独立的视口相对高度上限（如 `max-h-[60vh]`），应让 flex 分配高度生效；百分比高度（`h-full`）在 flex-grow 父级上不生效，需用 `flex-1` + `min-h-0` 传递高度约束。

### 代码一致性原则（重要）
- 当同一数据结构在不同地方渲染时（如表头 + 单元格、表单 + 预览等）：
  - 必须同时查看所有渲染位置，不能只改一处
  - 修改前列出对应关系，确认两边列数/结构一致
  - 判断逻辑必须统一抽离，不能各处各自实现
  - 禁止补丁式修复，应该一次性重写保证一致
  - 修改后验证：渲染位置A的列数 == 渲染位置B的列数
- 反例：
  - 表头改了显示规则，单元格没同步 → 数据错位
  - 同一判断逻辑多处实现，结果不一致 → 显示混乱
  - 每次只改一处打补丁 → 代码臃肿、bug累积
